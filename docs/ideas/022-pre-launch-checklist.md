# Idea 022: Pre-launch Checklist — Legal, Security, and Administrative

## Purpose

Gleisner のサービスローンチ前に完了すべき、**実装以外**のタスクを包括的にまとめたチェックリスト。
法務・行政手続き・セキュリティ運用・インフラ・ブランド保護のカテゴリに分類する。

実装タスクは各 ADR のチェックリストで管理する。ここでは「コードを書く以外にやること」に集中する。

## 2. 行政手続き・届出

### 2-A. 電気通信事業届出（ADR 022）

- [ ] 電気通信事業届出書（様式第8）を作成
- [ ] ネットワーク構成図を作成（Railway, Cloudflare, R2 の構成）
- [ ] 管轄の総合通信局を特定（事業所所在地ベース）
- [ ] 届出を提出
- [ ] 受付印付きの控えを保管

### 2-B. DMCA Designated Agent 登録（ADR 018）

- [ ] 米国著作権局に Designated Agent を電子登録（$6）
- [ ] `/legal/copyright` ページにエージェント情報を掲載
- [ ] 3 年後の更新リマインダーを設定

### 2-C. 事業体の決定

- [ ] 個人事業主 vs 法人化の判断（弁護士・税理士と相談）
  - 電気通信事業届出が個人で可能か確認（LC-022-5）
  - 税務上のメリット比較
  - 法人化する場合の設立手続き

### 2-D. 暗号技術の輸出規制確認

- [ ] Ed25519（楕円曲線暗号）の該非判定を確認
  - オープンソースライブラリ（libsodium 等）使用 → 公知技術として除外の可能性高い
  - 念のため経済産業省の安全保障貿易管理課に確認推奨

## 3. ブランド・ドメイン保護

- [ ] **商標登録出願**: 「Gleisner」を特許庁に出願
  - 対象区分: 第9類（プログラム）、第42類（ソフトウェア提供）、第45類（SNS 提供）
  - 費用: 出願料 3,400円 + 8,600円 × 区分数 + 弁理士費用
  - 審査期間: 約 6〜7 ヶ月 → **早期に出願すべき**
- [ ] ドメイン確保の確認（gleisner.app 等）
- [ ] ソーシャルアカウント確保（GitHub, X, etc.）

## 4. セキュリティ運用体制（実装以外）

### 4-A. インシデント対応計画（IRP）

- [ ] インシデント対応手順書を作成（NIST SP 800-61 ベース）
  - 準備 → 検知 → 封じ込め → 対処 → 復旧 → 振り返り
- [ ] データ漏洩時の通知フロー策定
  - 日本 APPI: 速報 3〜5 日以内 → 確報 30 日以内（個人情報保護委員会）
  - GDPR: 72 時間以内に監督当局へ通知
  - 本人への通知手順
- [ ] 緊急連絡先リスト作成（弁護士、インフラ事業者、セキュリティ専門家）

### 4-B. セキュリティ監査

- [ ] ローンチ前にペネトレーションテスト実施（外部専門家 or セルフ）
  - 最低限: OWASP Top 10 + GraphQL 固有の脆弱性チェック
  - 予算に応じて外部の診断サービス利用を検討
- [ ] 依存パッケージの脆弱性スキャン有効化（GitHub Dependabot / Snyk）

### 4-C. バックアップ・復旧

- [ ] PostgreSQL の自動バックアップ設定確認（Railway のバックアップポリシー確認）
- [ ] バックアップからの復旧手順を文書化・テスト
- [ ] Cloudflare R2 のメディアデータのバックアップ戦略決定

## 5. メール送信準備

通知メール（パスワードリセット、DMCA 通知等）を送信する場合:

- [ ] メール送信サービス選定（SendGrid, Amazon SES, Resend 等）

- [ ] DKIM 署名設定

- [ ] 特定電子メール法対応

  - 配信停止リンクの設置

- [ ] サイバー保険の見積もり取得・加入検討

## 8. ADR 020 へ追加すべきセキュリティギャップ

調査で発見された、ADR 020 に未記載のセキュリティ項目:

### ローンチ前 MUST

| カテゴリ | 項目 | 実装コスト |
|---------|------|-----------|
| HTTP ヘッダー | CSP, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy | 低 |
| 認証 | アカウント列挙防止（エラーメッセージ統一） | 低 |
| 認証 | ブルートフォース保護（ログイン試行制限） | 中 |
| 認証 | パスワードリセットトークンの有効期限・ワンタイム使用 | 低 |
| GraphQL | イントロスペクション無効化確認（本番） | 低（実装済みの可能性） |
| GraphQL | BOLA（オブジェクトレベル認可）の全 resolver 検証 | 中 |
| アップロード | ファイルタイプ検証（マジックバイト）、サイズ制限、EXIF 除去 | 中 |
| ログ | セキュリティイベントログ（ログイン成功/失敗、権限変更） | 中 |
| CORS | 本番で `*` 禁止の明示（現状コードは対応済みだが ADR に未記載） | 低 |

### ローンチ前 SHOULD

| カテゴリ | 項目 | 実装コスト |
|---------|------|-----------|
| 認証 | breached password チェック（HaveIBeenPwned API） | 低 |
| アップロード | マルウェアスキャン（ClamAV 等） | 中 |
| 運用 | 依存パッケージ脆弱性スキャン CI 組み込み | 低 |
| XSS | 出力エンコーディングの明示的検証 | 低（Hono + GraphQL で概ね安全だが確認） |
| SSRF | URL 入力のバリデーション（リンクプレビュー等実装時） | 低 |
| メール | SPF / DKIM / DMARC 設定 | 低 |

## 9. 将来の規制対応（ローンチ後に監視）

ローンチ時点では不要だが、成長に応じて対応が必要になる項目:

| トリガー | 対応 |
|---------|------|
| **月間ユーザー 1,000 万超** | 電気通信事業の「大規模事業者」指定 → 登録制に変更 |
| **月間発信者 1,000 万超 or 月間投稿 200 万超** | 情プラ法の大規模プラットフォーム義務（14 日以内対応、透明性レポート等） |
| **課税売上 1,000 万超** | 消費税の課税事業者に（翌々年から） |
| **投げ銭・チップ機能追加** | 資金移動業登録 or 前払式支払手段の検討が必要 |
| **アプリ内通貨の未使用残高 1,000 万超** | 前払式支払手段の届出 + 供託義務 |
| **有料サブスクリプション追加** | 特定商取引法に基づく表記義務 |
| **EU 市場で 3 年超 or 売上 €10M 超** | EU Copyright Directive Art.17 の小規模免除終了 |
| **オーストラリア進出** | Social Media Minimum Age Act 対応 |
